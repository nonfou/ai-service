# AI 聊天服务模块

## 📖 概述

AI 聊天服务模块是系统的核心功能,提供 OpenAI 兼容的聊天 API 接口,支持流式和非流式两种响应模式。本模块通过代理模式集成 Copilot 和 OpenRouter 两大后端服务,实现统一的 API 接口。

## 🎯 核心功能

- **OpenAI 兼容接口**: 完全兼容 OpenAI Chat Completions API
- **流式响应**: 基于 SSE (Server-Sent Events) 的实时流式输出
- **非流式响应**: 完整响应一次性返回
- **多模型支持**: 支持 GPT、Claude、Gemini 等多种 AI 模型
- **Token 估算**: 请求前估算 Token 数量
- **精确计费**: Token 级别的精确计费
- **会话粘性**: 保持同一会话使用同一后端账户
- **请求编排**: 完整的请求处理流程控制
- **错误处理**: 统一的异常处理和错误响应

## 📡 API 接口

### 聊天完成接口

**端点**: `POST /api/v1/chat/completions`

**认证方式**: API Key (Header: `Authorization: Bearer sk-xxxxx`)

#### 请求参数

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| model | String | 是 | 模型标识,如 `gpt-4o` |
| messages | Array | 是 | 消息列表 |
| temperature | Float | 否 | 随机性,0-2,默认 0.7 |
| max_tokens | Integer | 否 | 最大输出 Token 数 |
| top_p | Float | 否 | 核采样,0-1,默认 1 |
| stream | Boolean | 否 | 是否流式输出,默认 false |
| presence_penalty | Float | 否 | 存在惩罚,-2 到 2 |
| frequency_penalty | Float | 否 | 频率惩罚,-2 到 2 |

#### 消息格式

消息对象包含以下字段:
- **role**: 角色类型,可选值为 user(用户)、assistant(助手)、system(系统)
- **content**: 消息内容文本

#### 请求示例说明

**非流式请求特点:**
- 设置 `stream: false` 或省略该参数
- 包含完整的消息历史
- 可以设置 system 角色消息作为系统提示
- 支持设置 temperature、max_tokens 等参数

**流式请求特点:**
- 设置 `stream: true`
- 响应为 SSE (Server-Sent Events) 格式
- 实时返回生成的文本片段
- 用户可以看到逐字生成效果

#### 响应格式

**非流式响应结构:**
- id: 聊天完成的唯一标识
- object: 对象类型,值为 "chat.completion"
- created: Unix 时间戳
- model: 使用的模型名称
- choices: 选项数组,包含生成的消息
  - index: 选项索引(通常为0)
  - message: 消息对象,包含 role 和 content
  - finish_reason: 完成原因(stop/length/content_filter)
- usage: Token 使用统计
  - prompt_tokens: 输入 Token 数
  - completion_tokens: 输出 Token 数
  - total_tokens: 总 Token 数

**流式响应格式 (SSE):**
- 每个数据块以 `data: ` 开头
- 每个块是一个 JSON 对象
- object 类型为 "chat.completion.chunk"
- choices 中包含 delta 对象(而非 message)
- 第一个块通常包含 role
- 后续块包含 content 片段
- 最后一个块包含 finish_reason 和 usage
- 流结束时发送 `data: [DONE]`

## 🔄 请求处理流程

### 完整流程图

请求处理包含以下步骤:

**[1] API Key 认证**
- 验证 API Key 有效性
- 从数据库加载用户信息
- 检查用户状态是否正常

**[2] 参数验证**
- 验证必填参数(model, messages)
- 验证参数类型和格式
- 验证参数范围(如 temperature 0-2)

**[3] 模型验证**
- 检查模型是否存在
- 检查模型是否可用
- 获取模型配置信息

**[4] Token 估算**
- 估算输入 Token 数量
- 预估输出 Token 数量
- 计算预估费用

**[5] 余额检查**
- 查询用户当前余额
- 比较余额与预估费用
- 余额不足时返回错误

**[6] 账户选择**
- 根据会话 ID 选择账户(粘性路由)
- 或通过轮询算法选择可用账户
- 检查账户状态和配额

**[7] 请求代理**
- 将请求转发到后端服务(Copilot/OpenRouter)
- 处理流式或非流式响应
- 实时转发响应数据

**[8] Token 计数**
- 从响应中提取准确的 Token 数量
- 或使用自定义计数器统计实际 Token

**[9] 费用计算**
- 根据模型定价计算实际费用
- 应用缓存折扣(如有)
- 应用用户专属折扣(如有)

**[10] 余额扣减**
- 从用户余额扣除实际费用
- 使用分布式锁保证并发安全
- 记录余额变动日志

**[11] 调用记录**
- 保存完整的调用记录
- 包含请求参数、响应数据、费用等
- 用于统计分析和问题排查

**[12] 返回响应**
- 流式: 实时返回 SSE 数据流
- 非流式: 返回完整 JSON 响应
- 发生错误时返回标准错误格式

### 错误处理流程

**认证失败:**
- 返回 401 Unauthorized
- 错误信息: "Invalid API key"

**参数验证失败:**
- 返回 400 Bad Request
- 错误信息: 具体的参数错误说明

**模型不存在:**
- 返回 400 Bad Request
- 错误信息: "Model not found"

**余额不足:**
- 返回 402 Payment Required
- 错误信息: "Insufficient balance"

**后端服务异常:**
- 返回 500 Internal Server Error
- 错误信息: 后端服务返回的错误
- 记录详细错误日志

**超时:**
- 返回 504 Gateway Timeout
- 错误信息: "Request timeout"

## 🧩 核心组件

### 1. ChatController

**职责**: HTTP 请求处理和响应

**主要端点:**

#### POST /api/v1/chat/completions - 聊天完成接口

处理流程:
1. 从请求头提取 API Key
2. 解析请求体为 ChatRequest 对象
3. 验证参数有效性
4. 调用 ChatWorkflowService 处理请求
5. 根据 stream 参数返回流式或非流式响应

流式响应处理:
- 设置 Content-Type 为 text/event-stream
- 设置 Cache-Control 为 no-cache
- 返回 SseEmitter 对象
- 监听后端服务的 SSE 流
- 逐块转发给客户端
- 结束时发送 [DONE] 标记

非流式响应处理:
- 等待完整响应
- 构造 ChatCompletion 对象
- 返回 JSON 格式响应

异常处理:
- 捕获所有异常
- 转换为标准错误格式
- 记录错误日志
- 流式响应中断时关闭 SSE 连接

### 2. ChatWorkflowService

**职责**: 编排聊天请求的完整处理流程

**主要方法:**

#### processChat() - 处理聊天请求

流程编排:
1. 验证 API Key 并获取用户信息
2. 验证模型存在性和可用性
3. 估算输入 Token 数量
4. 检查用户余额是否充足
5. 选择合适的后端账户
6. 调用模型代理服务
7. 处理响应并计算实际费用
8. 扣减用户余额
9. 保存调用记录
10. 返回结果

#### processChatStream() - 处理流式聊天

特殊处理:
- 创建 SseEmitter 对象
- 异步处理后端响应流
- 实时计数 Token
- 流结束后统一扣费和记录
- 异常时确保资源释放

#### selectAccount() - 选择后端账户

选择策略:
- 优先使用会话粘性(同一 conversation_id 使用同一账户)
- 会话不存在时使用轮询算法
- 只选择状态为 active 的账户
- 检查账户配额未超限
- 记录账户使用情况

#### calculateCost() - 计算费用

计费逻辑:
- 获取模型定价配置
- 分别计算输入和输出费用
- 应用缓存折扣(prompt caching)
- 应用用户专属折扣
- 返回最终费用

### 3. ModelProxy 接口

**职责**: 定义模型代理的统一接口

**接口方法:**

#### chat() - 非流式聊天

输入参数:
- account: 后端账户信息
- request: 聊天请求对象
- model: 模型配置信息

返回结果:
- ChatCompletion 对象
- 包含完整响应和 Token 统计

#### chatStream() - 流式聊天

输入参数:
- account: 后端账户信息
- request: 聊天请求对象
- model: 模型配置信息
- emitter: SSE 发射器

处理方式:
- 异步处理响应流
- 实时发送数据块到 emitter
- 累积计算 Token 数量
- 流结束时返回 Token 统计

#### isSupported() - 检查是否支持指定模型

用途:
- 判断代理是否支持某个模型
- 用于路由选择

### 4. CopilotProxyService

**职责**: 实现 Copilot 后端的代理逻辑

**特点:**
- 支持 GPT 系列模型
- 通过 GitHub Copilot API 调用
- 需要 Copilot 账户认证
- 支持流式和非流式

**实现细节:**

#### 请求转换
- 将标准 ChatRequest 转换为 Copilot 格式
- 设置必要的请求头
- 添加认证信息
- 映射模型名称

#### 响应处理
- 解析 Copilot 响应格式
- 转换为标准 ChatCompletion 格式
- 提取 Token 统计信息
- 处理错误响应

#### 流式处理
- 建立 SSE 连接到 Copilot
- 逐块解析 JSON 数据
- 转换为标准流式格式
- 实时转发给客户端

### 5. OpenRouterProxyService

**职责**: 实现 OpenRouter 后端的代理逻辑

**特点:**
- 支持更多模型类型(Claude, Gemini 等)
- 通过 OpenRouter API 调用
- 需要 OpenRouter API Key
- 标准 OpenAI 兼容格式

**实现方式:**
- 请求格式已兼容,直接转发
- 添加 OpenRouter 认证信息
- 设置 HTTP-Referer 头
- 处理 OpenRouter 特有的响应字段

### 6. TokenEstimator

**职责**: 估算和计数 Token 数量

**方法:**

#### estimateTokens() - 估算输入 Token

算法:
- 对于 GPT 模型:使用 tiktoken 库精确计算
- 对于其他模型:按字符数/4 粗略估算
- 考虑消息格式的额外开销
- 支持多轮对话的累积计算

#### countStreamTokens() - 计数流式输出 Token

实时计数:
- 累积所有响应块的文本
- 流结束后统一计算
- 使用相同的估算算法
- 返回准确的 Token 数

#### 估算准确性
- GPT 模型: 95%以上准确度
- Claude 模型: 80-90%准确度
- 其他模型: 70-80%准确度
- 主要用于预检查,实际计费使用响应中的 Token 数

## 💰 计费说明

### 计费项目

**输入 Token 计费:**
- 按实际输入的 Token 数量计费
- 包含所有历史消息
- 包含 system 提示

**输出 Token 计费:**
- 按实际生成的 Token 数量计费
- 输出价格通常高于输入

**Prompt Caching 折扣:**
- 重复的提示内容享受缓存折扣
- 折扣力度: 50-90%
- 需要模型支持

### 计费公式

**基础费用计算:**
- 输入费用 = 输入Token数 × 输入单价 / 1000
- 输出费用 = 输出Token数 × 输出单价 / 1000
- 总费用 = 输入费用 + 输出费用

**应用折扣:**
- 缓存折扣后费用 = 缓存Token数 × 输入单价 × 缓存折扣率 / 1000
- 最终费用 = (输入费用 - 缓存折扣后费用) + 输出费用

**用户折扣:**
- 如果用户有专属折扣
- 最终费用 = 最终费用 × 用户折扣率

### 价格示例

**GPT-4o 模型定价 (示例):**
- 输入: $0.0025 / 1K tokens
- 输出: $0.010 / 1K tokens
- 缓存折扣: 50%

**计费示例:**
- 输入 1000 tokens
- 输出 500 tokens
- 无缓存: $0.0025 + $0.005 = $0.0075
- 有缓存 500 tokens: $0.00125 + $0.005 = $0.00625

## 🔍 常见问题

### Q: 为什么选择 OpenAI 兼容格式?

A: OpenAI API 格式是业界事实标准,具有以下优势:

**生态丰富:**
- 大量第三方库和工具支持
- 开发者熟悉度高
- 文档和示例丰富

**兼容性好:**
- 可以无缝替换 OpenAI 服务
- 支持现有的 OpenAI 客户端代码
- 降低迁移成本

**标准化:**
- 接口设计合理
- 参数命名清晰
- 错误格式统一

### Q: 流式和非流式有什么区别?

A: 两种模式各有特点和适用场景:

**流式响应 (stream=true):**

优点:
- 用户体验好,可以实时看到生成过程
- 延迟低,首字显示快
- 适合对话场景

缺点:
- 实现复杂度高
- 需要处理 SSE 连接
- 异常处理更困难

适用场景:
- 聊天机器人
- 交互式对话
- 长文本生成

**非流式响应 (stream=false):**

优点:
- 实现简单
- 易于处理和解析
- 适合批处理

缺点:
- 等待时间长
- 用户体验较差
- 不适合长内容

适用场景:
- API 集成
- 批量处理
- 短文本生成

### Q: Token 估算准确吗?

A: Token 估算的准确度取决于模型:

**GPT 系列模型:**
- 使用 tiktoken 库精确计算
- 准确度 95%以上
- 几乎等同于实际消耗

**Claude 系列模型:**
- 使用近似算法
- 准确度 80-90%
- 可能有小幅偏差

**其他模型:**
- 使用字符数估算
- 准确度 70-80%
- 主要用于预检查

**建议:**
- 估算用于预检查余额是否充足
- 实际计费使用响应中的准确 Token 数
- 预留一定余量避免余额不足

### Q: 如何处理超时?

A: 系统对超时有多层处理机制:

**客户端超时:**
- 建议设置 60-120 秒超时
- 长文本生成可以适当延长
- 流式响应建议设置更长超时

**服务端超时:**
- 后端请求默认超时 60 秒
- 可以通过配置调整
- 超时后返回 504 错误

**处理建议:**
- 客户端实现重试机制
- 指数退避重试策略
- 记录超时日志便于排查
- 考虑拆分大请求

### Q: 支持多轮对话吗?

A: 完全支持多轮对话:

**实现方式:**
- 将历史消息放入 messages 数组
- 按时间顺序排列
- 每次请求包含完整历史

**示例��构:**
- 第一轮: [user: "你好"]
- 第二轮: [user: "你好", assistant: "你好!", user: "今天天气怎么样?"]
- 第三轮: [历史消息..., user: "谢谢"]

**注意事项:**
- 历史消息越多,Token 消耗越大
- 建议限制历史消息数量(如最近10轮)
- 可以使用总结机制压缩历史
- 超长历史可能触发 Token 限制

**会话粘性:**
- 同一 conversation_id 会路由到同一账户
- 保证对话连贯性
- 提升缓存命中率

### Q: 如何处理敏感内容?

A: 系统有多层内容安全机制:

**模型自带过滤:**
- GPT/Claude 等模型内置内容过滤
- 自动拒绝不当请求
- 返回 content_filter 错误

**自定义过滤(可选):**
- 可以实现自定义敏感词过滤
- 请求前检查输入内容
- 拒绝包含敏感词的请求

**日志脱敏:**
- 记录日志时脱敏敏感信息
- 保护用户隐私
- 符合合规要求

**建议:**
- 不要用于生成违法内容
- 遵守服务条款
- 客户端也应实现过滤

### Q: 并发限制是多少?

A: 并发限制取决于多个因素:

**用户级限制:**
- 可以为不同用户设置不同限制
- 默认建议 10-50 并发
- 超过限制返回 429 错误

**系统级限制:**
- 由后端账户数量决定
- 每个账户支持一定并发
- 总并发 = 账户数 × 单账户并发

**优化建议:**
- 客户端实现请求队列
- 控制并发数量
- 实现失败重试
- 使用连接池

**监控:**
- 实时监控并发数
- 设置告警阈值
- 及时扩容后端账户

## 📊 性能优化

### 1. 连接池配置

**HTTP 连接池:**
- 使用连接池复用连接
- 减少 TCP 握手开销
- 建议配置:
  - 最大连接数: 200
  - 每个路由最大连接: 50
  - 连接超时: 5秒
  - 读超时: 60秒

**数据库连接池:**
- HikariCP 连接池
- 建议配置:
  - 最小空闲连接: 10
  - 最大连接数: 50
  - 连接超时: 30秒
  - 空闲超时: 600秒

### 2. 异步处理

**流式响应异步化:**
- 使用异步线程处理后端流
- 主线程立即返回 SSE 连接
- 避免阻塞 HTTP 线程池

**调用记录异步化:**
- 保存调用记录使用异步
- 不影响响应速度
- 失败时记录日志

**余额扣减同步化:**
- 余额扣减必须同步完成
- 使用分布式锁保证一致性
- 失败时回滚或重试

### 3. 缓存策略

**模型配置缓存:**
- 缓存模型列表和定价
- 更新频率低,缓存时间长(1小时)
- 减少数据库查询

**用户信息缓存:**
- 缓存用户基本信息
- 更新时主动失效
- 缓存时间 5-10 分钟

**API Key 缓存:**
- 缓存 API Key 到用户的映射
- 减少数据库查询
- 缓存时间 10 分钟

**不建议缓存:**
- 用户余额(频繁变动)
- 调用记录(实时性要求高)

### 4. 数据库优化

**索引优化:**
- API Key 索引
- 用户 ID 索引
- 模型名称索引
- 调用时间索引

**查询优化:**
- 只查询必要字段
- 避免 N+1 查询
- 使用批量查询

**连接优化:**
- 使用连接池
- 合理设置超时
- 监控慢查询

## 🔒 安全建议

### 1. API Key 安全

**生成规则:**
- 使用强随机算法
- 长度建议 32-64 字符
- 包含字母、数字、特殊字符

**存储方式:**
- 数据库存储哈希值
- 不存储明文
- 使用 bcrypt 或 SHA-256

**使用建议:**
- 定期轮换 API Key
- 不同场景使用不同 Key
- 妥善保管,不要泄露
- 发现泄露立即撤销

### 2. 速率限制

**按用户限制:**
- 每分钟请求数限制
- 每小时请求数限制
- 每天请求数限制

**按 IP 限制:**
- 防止单 IP 过度请求
- 识别异常流量
- 自动封禁恶意 IP

**实现方式:**
- 使用 Redis 计数器
- 滑动窗口算法
- 令牌桶算法

### 3. 输入验证

**参数验证:**
- 验证必填参数
- 验证参数类型
- 验证参数范围
- 防止 SQL 注入

**内容验证:**
- 长度限制
- 格式检查
- 敏感词过滤
- XSS 防护

### 4. 日志脱敏

**敏感信息脱敏:**
- API Key 脱敏(只记录前后几位)
- 用户消息内容选择性记录
- 个人信息脱敏
- 支付信息脱敏

**日志级别:**
- 生产环境使用 INFO 级别
- 调试时使用 DEBUG
- 错误详细记录

**日志保留:**
- 根据合规要求设置保留期
- 定期清理旧日志
- 归档重要日志

**请求指标:**

- 请求响应时间
- 请求成功率
- 错误率分布

**业务指标:**
- Token 消耗量
- 费用统计
- 活跃用户数
- 模型使用分布

**性能指标:**
- CPU 使用率
- 内存使用率
- 数据库连接数
- 缓存命中率

**可用性指标:**
- 系统可用率
- 后端账户可用率
- API 响应成功率

### 监控查询

**实时监控:**
- 使用 Prometheus 采集指标
- Grafana 展示监控面板
- 设置告警规则

**日志监控:**
- ELK 栈收集和分析日志
- 设置关键字告警
- 错误日志实时推送

**业务监控:**
- 统计每日调用量
- 统计收入和成本
- 识别异常用户

- 错误率 > 5% 告警
- 响应时间 > 5秒 告警

- 余额不足告警
