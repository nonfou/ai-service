# 订阅套餐模块

## 📖 概述

订阅套餐模块提供灵活的套餐管理和订阅功能,用户可以购买套餐获得额度和特权。

## 🎯 核心功能

- 套餐管理(CRUD)
- 套餐购买和订阅
- 自动额度发放
- 订阅历史记录
- 套餐展示和排序
- 套餐状态管理

## 📦 套餐设计

### 套餐结构

订阅套餐包含以下核心字段:
- **planName**: 套餐标识(如trial_card, max_100)
- **displayName**: 显示名称(如"体验卡", "Max 100")
- **description**: 套餐说明
- **originalPrice**: 原价
- **price**: 现价(实际售价)
- **quotaAmount**: 额度金额(购买后获得的余额)
- **features**: 功能特性列表(JSON格式)
- **colorTheme**: 颜色主题(用于前端展示)
- **badgeText**: 徽章文字(如"新用户专享", "推荐套餐")
- **status**: 状态(1-上架 0-下架)
- **sortOrder**: 排序值(数字越小越靠前)

## 🔄 订阅流程

### 购买流程

完整的套餐购买流程包括以下步骤:

1. **用户选择套餐**: 用户浏览套餐列表并选择合适的套餐
2. **检查用户余额**: 验证用户账户余额是否充足
3. **扣除套餐费用**: 从用户余额中扣除套餐价格
4. **创建订阅记录**: 记录订阅信息(套餐、金额、有效期等)
5. **发放套餐额度**: 自动向用户账户充值对应额度
6. **记录余额变动**: 记录扣费和充值的余额变动日志
7. **更新用户余额**: 更新用户最新余额状态
8. **返回订阅信息**: 返回订阅成功信息给用户

### 订阅状态

订阅记录有三种状态:
- **active**: 生效中,当前正在使用的订阅
- **expired**: 已过期,超过有效期的订阅
- **cancelled**: 已取消,用户主动取消的订阅

### 自动过期处理

系统通过定时任务自动处理过期订阅:
- 执行时间: 每天凌晨1点
- 处理逻辑: 查询状态为active但已超过endDate的订阅
- 批量更新: 将这些订阅的状态更新为expired
- 日志记录: 记录每个过期订阅的用户ID和套餐名称

## 🔧 核心组件

### 1. SubscriptionController

**用户接口**,提供以下功能:

- **GET /api/subscriptions/plans**: 获取套餐列表
  - 返回所有上架的套餐
  - 按sortOrder排序
  - 包含套餐详情和特性

- **POST /api/subscriptions/subscribe**: 订阅套餐
  - 接收套餐ID
  - 验证用户余额
  - 执行订阅流程
  - 返回订阅结果

- **GET /api/subscriptions/history**: 订阅历史
  - 支持分页查询
  - 显示用户所有订阅记录
  - 包含套餐信息和状态

- **POST /api/subscriptions/{id}/cancel**: 取消订阅
  - 取消指定订阅
  - 更新状态为cancelled
  - 验证订阅所有权

- **GET /api/subscriptions/current**: 当前订阅
  - 查询用户当前有效订阅
  - 返回最新的active状态订阅

### 2. SubscriptionService

**核心服务**,主要方法:

#### subscribe() - 订阅套餐

执行完整的订阅流程:
1. 查询并验证套餐信息
2. 检查用户余额是否充足
3. 调用BalanceService扣除套餐费用
4. 调用BalanceService发放套餐额度
5. 创建订阅记录并保存
6. 计算订阅有效期
7. 返回订阅信息

异常处理:
- 套餐不存在或已下架
- 用户余额不足
- 数据库操作失败

#### cancelSubscription() - 取消订阅

取消用户订阅:
1. 查询订阅记录
2. 验证订阅存在性
3. 验证用户权限
4. 验证订阅状态(只能取消active状态)
5. 更新状态为cancelled
6. 记录操作日志

注意事项:
- 已发放的额度不会回收
- 取消操作不可逆
- 只能取消自己的订阅

#### getCurrentSubscription() - 获取当前订阅

查询用户当前有效订阅:
- 筛选条件: userId + status=active + endDate>=today
- 按创建时间倒序
- 只返回最新的一条记录
- 没有有效订阅时返回null

#### calculateEndDate() - 计算结束日期

根据套餐特性计算有效期:
- 从features字段提取天数(如"7天有效期")
- 支持的天数: 7天、30天、90天、365天
- 默认值: 30天
- 返回: 当前日期 + 天数

### 3. AdminPlanController

**管理员接口**,提供以下功能:

- **GET /api/admin/plans**: 获取所有套餐
  - 包含已下架的套餐
  - 管理员可见所有数据

- **POST /api/admin/plans**: 创建套餐
  - 设置套餐所有字段
  - 自动生成时间戳
  - 需要admin权限

- **PUT /api/admin/plans/{id}**: 更新套餐
  - 修改套餐信息
  - 可更新价格、额度、特性等
  - 记录操作日志

- **DELETE /api/admin/plans/{id}**: 删除套餐
  - 物理删除套餐记录
  - 谨慎操作,建议使用下架代替

- **PUT /api/admin/plans/{id}/status**: 上架/下架套餐
  - 切换status字段
  - 下架后用户不可见
  - 不影响已订阅用户

- **PUT /api/admin/plans/{id}/sort**: 调整排序
  - 修改sortOrder值
  - 控制前端展示顺序
  - 数字越小越靠前

## 📊 数据模型

### SubscriptionPlan 实体

套餐实体,存储套餐基本信息:
- id: 主键ID
- planName: 套餐标识
- displayName: 显示名称
- description: 套餐说明
- originalPrice: 原价
- price: 现价
- quotaAmount: 额度金额
- features: 功能特性(JSON字符串)
- colorTheme: 颜色主题
- badgeText: 徽章文字
- status: 状态
- sortOrder: 排序
- createdAt: 创建时间
- updatedAt: 更新时间

### Subscription 实体

订阅记录实体,记录用户订阅历史:
- id: 主键ID
- userId: 用户ID
- planId: 套餐ID
- planName: 套餐名称(冗余字段,防止套餐删除后无法查看)
- amount: 支付金额
- quotaAmount: 获得额度
- startDate: 开始日期
- endDate: 结束日期
- status: 订阅状态(active/expired/cancelled)
- createdAt: 创建时间
- updatedAt: 更新时间

数据冗余设计:
- planName字段冗余存储,防止套餐删除后历史记录丢失信息
- amount和quotaAmount记录当时的价格,不受后续套餐调整影响

## 💻 前端集成

### 获取套餐列表

前端通过以下方式获取套餐列表:
- 接口: GET /api/subscriptions/plans
- 请求头: Authorization: Bearer {token}
- 响应: 返回套餐数组,包含所有上架套餐
- 数据结构: 每个套餐包含价格、额度、特性等完整信息

### 订阅套餐

前端订阅套餐的流程:
- 接口: POST /api/subscriptions/subscribe
- 请求头: Authorization + Content-Type: application/json
- 请求体: { "planId": 套餐ID }
- 成功响应: code=200,返回订阅信息
- 失败响应: code!=200,message字段包含错误信息

错误处理:
- 余额不足: 提示用户充值
- 套餐不存在: 刷新套餐列表
- 网络错误: 显示友好提示

## 🔍 常见问题

### Q: 套餐过期后余额会消失吗?

A: 不会。套餐购买后立即发放的额度会直接加到账户余额中,不会因套餐过期而消失。

套餐有效期的含义:
- 有效期只是订阅记录的时间范围
- 额度在订阅时一次性发放到余额
- 余额是永久有效的(除非另有过期规则)
- 过期只是将订阅状态从active变为expired

### Q: 可以同时购买多个套餐吗?

A: 可以。每次购买都会创建新的订阅记录并发放额度。

多次购买的处理:
- 每次购买独立扣费和发放额度
- 额度会累加到账户余额
- 多个订阅记录并行存在
- getCurrentSubscription返回最新的有效订阅

### Q: 套餐可以退款吗?

A: 默认不支持退款。取消订阅只是更新状态,已发放的额度不会回收。

如需支持退款:
- 需要额外开发退款接口
- 检查额度是否已使用
- 计算可退款金额
- 回收未使用额度
- 退款到用户余额或原支付方式

### Q: 如何自定义套餐有效期?

A: 在套餐的features字段中添加"XX天有效期"的描述,系统会自动解析。

支持的格式:
- "7天有效期"  → 7天
- "30天有效期" → 30天
- "90天有效期" → 90天
- "365天有效期" → 365天
- 不包含天数描述时默认30天

系统通过字符串匹配自动识别天数。

## 📈 统计分析

### 套餐销售统计

分析各套餐的销售情况:
- 统计维度: 按planName分组
- 销售数量: 订阅记录总数
- 总收入: amount字段求和
- 平均价格: 总收入 / 销售数量
- 筛选条件: status为active或expired(排除cancelled)

应用场景:
- 识别热门套餐
- 评估定价合理性
- 优化套餐组合

### 每日订阅趋势

分析订阅趋势:
- 统计维度: 按日期分组
- 订阅数量: 每日新增订阅数
- 每日收入: 每日amount求和
- 时间范围: 最近30天或自定义

数据价值:
- 了解用户购买行为
- 识别营销活动效果
- 预测未来收入

### 用户订阅行为分析

分析用户订阅特征:
- 首次订阅用户数
- 重复订阅用户数
- 套餐升级路径
- 平均订阅周期

业务价值:
- 优化用户引导流程
- 设计套餐升级策略
- 提高用户留存率

## 🚀 未来扩展

### 自动续费

自动续费功能实现思路:

**数据模型扩展**:
- Subscription表添加autoRenew字段(布尔类型)
- 用户订阅时可选择是否自动续费

**定时任务**:
- 执行时间: 每天凌晨2点
- 查询条件: status=active + endDate=今天+3天 + autoRenew=true
- 处理逻辑: 尝试自动订阅相同套餐
- 异常处理: 余额不足时发送续费失败通知

**用户体验**:
- 提前3天尝试续费,给用户准备时间
- 续费失败发送邮件/站内通知
- 用户可随时关闭自动续费

### 优惠券系统

优惠券功能实现思路:

**优惠券类型**:
- 折扣券: 按比例折扣(如9折)
- 立减券: 固定金额优惠(如减10元)
- 满减券: 满XX元减YY元

**使用流程**:
1. 用户输入优惠券码
2. 验证优惠券有效性(状态、有效期、使用次数)
3. 检查适用范围(全部套餐或指定套餐)
4. 计算折扣后价格
5. 记录优惠券使用记录
6. 完成订阅

**数据模型**:
- Coupon表: 存储优惠券信息
- CouponUsage表: 记录使用历史
- 关联到Subscription表

### 套餐组合

套餐组合功能:
- 一次购买多个套餐享受优惠
- 设置组合价格和组合额度
- 适用于大批量采购场景

### 企业套餐

面向企业用户的特殊套餐:
- 支持团队账号管理
- 批量开通子账号
- 统一计费和额度分配
