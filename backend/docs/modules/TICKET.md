# 工单系统模块

## 📖 概述

工单系统模块提供完整的用户反馈和问题处理功能,支持工单创建、分配、处理、回复和关闭的全流程管理。

## 🎯 核心功能

- 工单创建与提交
- 工单分类管理
- 工单优先级设置
- 工单分配与转派
- 工单回复与沟通
- 工单状态跟踪
- 附件上传支持
- 工单评价反馈
- 工单统计分析
- 自动提醒通知

## 📦 工单设计

### 工单状态流转

工单的生命周期包含以下状态:

1. **创建(open)**: 用户刚创建工单,等待管理员处理
2. **待处理(pending)**: 工单已分配给管理员,等待开始处理
3. **处理中(processing)**: 管理员正在处理工单
4. **等待用户(waiting_user)**: 管理员已回复,等待用户反馈
5. **已解决(resolved)**: 问题已解决,等待用户确认
6. **已关闭(closed)**: 工单已完成并关闭
7. **已取消(cancelled)**: 用户取消工单

状态可以在 processing 和 waiting_user 之间来回切换,反映双方持续沟通的过程。任何状态都可以直接转为 cancelled 状态。

### 工单优先级

系统支持四个优先级级别:

- **紧急(urgent)**: P0级别,需立即处理,1小时内响应
- **高(high)**: P1级别,24小时内处理
- **中(medium)**: P2级别,3天内处理
- **低(low)**: P3级别,7天内处理

### 工单分类

支持以下工单类型:

- **账户问题**: 登录、注册、密码重置
- **充值问题**: 支付失败、到账延迟
- **使用问题**: API调用、功能使用
- **计费问题**: 扣费异常、余额问题
- **技术支持**: 集成咨询、技术疑问
- **功能建议**: 新功能建议、改进意见
- **其他**: 其他类型问题

## 📊 数据模型

### Ticket ��体

工单主表,存储工单基本信息:

**基本信息字段:**
- id: 主键ID
- ticketNo: 工单号(唯一标识,格式: TK + 日期 + 随机数)
- userId: 创建用户ID
- username: 用户名(冗余字段)
- email: 联系邮箱

**工单内容字段:**
- category: 分类
- subject: 标题
- description: 详细描述
- priority: 优先级
- status: 状态

**分配信息字段:**
- assignedTo: 分配给的管理员ID
- assignedToName: 分配给的管理员名称

**评价反馈字段:**
- rating: 用户评分(1-5分)
- feedback: 用户反馈文字

**时间记录字段:**
- firstResponseAt: 首次响应时间(用于SLA统计)
- resolvedAt: 解决时间
- closedAt: 关闭时间
- createdAt: 创建时间
- updatedAt: 更新时间

### TicketReply 实体

工单回复表,记录所有对话内容:

**回复者信息:**
- id: 主键ID
- ticketId: 所属工单ID
- userId: 回复用户ID
- username: 回复用户名
- isStaff: 是否为工作人员(区分用户和管理员)

**回复内容:**
- content: 回复内容
- attachments: 附件列表(JSON格式)
- internal: 是否为内部备注(仅管理员可见)

**时间戳:**
- createdAt: 创建时间
- updatedAt: 更新时间

### TicketAttachment 实体

工单附件表,记录上传的文件:

**关联信息:**
- id: 主键ID
- ticketId: 所属工单ID
- replyId: 所属回复ID(可选)
- userId: 上传用户ID

**文件信息:**
- fileName: 文件名
- fileType: 文件类型(MIME类型)
- fileSize: 文件大小(字节)
- filePath: 存储路径
- fileUrl: 访问URL

**时间戳:**
- createdAt: 上传时间

## 🔧 核心组件

### 1. TicketController

**用户接口**,提供以下功能:

#### POST /api/tickets - 创建工单
- 接收工单信息(分类、标题、描述、优先级、附件)
- 生成唯一工单号
- 自动获取用户信息
- 发送通知给管理员
- 返回创建的工单信息

#### GET /api/tickets/my - 获取我的工单列表
- 支持分页查询
- 支持按状态筛选
- 支持按分类筛选
- 按创建时间倒序排列
- 返回工单列表

#### GET /api/tickets/{id} - 获取工单详情
- 验证用户权限(只能查看自己的工单)
- 返回工单信息
- 返回所有回复记录(不含内部备注)
- 返回所有附件

#### POST /api/tickets/{id}/reply - 回复工单
- 验证工单所有权
- 验证工单状态(已关闭的不能回复)
- 创建回复记录
- 处理附件上传
- 自动更新工单状态
- 通知负责管理员

#### POST /api/tickets/{id}/close - 关闭工单
- 验证权限
- 更新状态为已关闭
- 记录关闭时间
- 可选提交评分和反馈
- 清除相关缓存

#### POST /api/tickets/{id}/cancel - 取消工单
- 验证所有权
- 验证状态(已关闭的不能取消)
- 更新状态为已取消
- 清除缓存

#### POST /api/tickets/{id}/reopen - 重新打开工单
- 验证所有权
- 验证状态(只能重开已关闭或已解决的)
- 清除解决时间和关闭时间
- 添加重开原因作为回复
- 通知负责管理员

#### POST /api/tickets/upload - 上传附件
- 验证文件大小(限制10MB)
- 验证文件类型(图片、PDF、Office文档等)
- 存储文件
- 返回文件URL

#### GET /api/tickets/stats/category - 获取分类统计
- 统计用户各分类的工单数量
- 用于展示数据概览

### 3. TicketService

**核心服务**,主要方法:

#### createTicket() - 创建工单

执行以下步骤:
1. 查询用户信息
2. 生成唯一工单号(格式: TK + yyyyMMdd + 6位随机数)
3. 创建工单记录
4. 保存附件信息
5. 发送通知给管理员
6. 清除用户工单列表缓存
7. 返回工单信息

#### getUserTickets() - 获取用户工单列表

查询条件:
- 按用户ID筛选
- 可选按状态筛选
- 可选按分类筛选
- 按创建时间倒序
- 支持分页

#### getTicketDetail() - 获取工单详情

返回信息:
- 工单基本信息
- 所有回复记录(普通用户不含内部备注)
- 所有附件列表

权限检查:
- 验证是否为工单创建者
- 管理员可查看所有工单

#### replyTicket() - 用户回复工单

处理流程:
1. 验证工单存在性和所有权
2. 验证工单状态(不能回复已关闭的)
3. 创建回复记录
4. 保存附件
5. 自动更新工单状态(waiting_user → processing)
6. 发送通知给负责管理员

#### adminReplyTicket() - 管理员回复工单

处理流程:
1. 验证工单存在性
2. 创建管理员回复(标记isStaff=true)
3. 保存附件
4. 更新首次响应时间(如果是第一次响应)
5. 自动更新工单状态:
   - open/pending → processing
   - processing → waiting_user
6. 发送通知给用户(非内部备注)

#### assignTicket() - 分配工单

处理流程:
1. 验证工单存在性
2. 验证指定管理员身份
3. 更新分配信息
4. 自动更新状态(open → pending)
5. 发送分配通知给管理员

#### updateTicketStatus() - 更新工单状态

处理流程:
1. 记录旧状态
2. 更新为新状态
3. 自动记录解决时间(状态变为resolved时)
4. 发送状态变更通知给用户

#### closeTicket() - 关闭工单

处理流程:
1. 验证权限
2. 更新状态为closed
3. 记录关闭时间
4. 保存评分和反馈(可选)
5. 清除缓存

#### cancelTicket() - 取消工单

处理流程:
1. 验证所有权
2. 验证当前状态
3. 更新状态为cancelled
4. 清除缓存

#### reopenTicket() - 重新打开工单

处理流程:
1. 验证所有权和状态
2. 更新状态为open
3. 清除解决时间和关闭时间
4. 添加重开原因作为回复
5. 通知负责管理员
6. 清除缓存

#### uploadAttachment() - 上传附件

处理流程:
1. 验证文件类型(支持图片、PDF、Word、Excel、文本)
2. 存储文件到指定路径
3. 生成访问URL
4. 返回文件信息

#### batchOperation() - 批量操作

支持操作类型:
- assign: 批量分配
- status: 批量更新状态
- priority: 批量更新优先级

处理方式:
- 循环处理每个工单
- 单个失败记录日志但不中断
- 保证部分成功可以生效

#### getStatsOverview() - 获取统计概览

统计指标:
- 总工单数
- 各状态工单数(open/pending/resolved/closed)
- 今日新增数
- 平均评分
- 平均首次响应时间(分钟)

#### getTicketTrend() - 获取工单趋势

查询指定日期范围内:
- 每日创建的工单数
- 每日关闭的工单数

用于前端绘制趋势图表。

#### getAdminWorkload() - 获取管理员工作量

统计各管理员的:
- 负责工单总数
- 活跃工单数(进行中的)
- 平均评分

用于工作量分析和平衡。

#### exportTickets() - 导出工单

处理流程:
1. 根据条件查询工单列表
2. 使用Excel工具类生成文件
3. 返回字节数组
4. 由Controller设置响应头下载

### 内部辅助方法

- **generateTicketNo()**: 生成唯一工单号
- **saveAttachments()**: 保存附件记录到数据库
- **clearTicketCache()**: 清除用户工单列表缓存

- **isAllowedFileType()**: 验证文件类型是否允许上传

## 🚀 性能优化

### 缓存策略

**工单列表缓存:**
- 缓存key: userTickets:{userId}:{status}
- 过期时间: 5分钟
- 更新时机: 创建、回复、关闭工单时清除

**工单详情缓存:**
- 不建议缓存详情,因为更新频繁
- 如需缓存,过期时间不超过1分钟

### 索引优化

**单列索引:**
- 工单号、用户ID、状态等常用查询字段

**复合索引:**
- (user_id, status, created_at DESC) - 用户筛选查询
- (status, priority, created_at DESC) - 管理员筛选查询
- (assigned_to, status, created_at DESC) - 工作量查询

### 查询优化

**分页查询优化:**
- 使用覆盖索引减少回表

**统计查询优化:**
- 使用Redis缓存统计结果

**附件查询优化:**
- attachments字段使用JSON类型

## 🔍 常见问题

### Q: 工单附件如何存储?

A: 支持两种存储方式:

**本地存储方式:**
- 存储在服务器磁盘的指定目录
- 通过Nginx配置静态文件访问
- 适用于小规模应用

**云存储方式:**
- 使用阿里云OSS或腾讯云COS
- 上传后获取公网URL
- 支持CDN加速
- 适用于大规模应用

推荐使用云存储,更稳定可靠。

### Q: 如何实现工单自动分配?

A: 可以实现自动分配规则:

**按类别分配:**
- 为每个分类配置默认负责人
- 新工单自动分配给对应管理员

**按工作量均衡:**
- 统计各管理员当前活跃工单数
- 分配给工作量最少的管理员

**轮询分配:**
- 按顺序依次分配给各管理员
- 使用Redis记录当前轮询位置

**智能分配:**
- 根据管理员专长和工单类型匹配
- 考虑历史评分和响应速度
- 使用机器学习优化分配策略

### Q: 如何处理超时未响应的工单?

A: 使用定时任务监控SLA违规:

**定时检查:**
- 每小时执行一次
- 查询超时未响应的工单
- 根据优先级设置不同超时时间

**超时处理:**
- 发送提醒通知给负责管理员
- 发送邮件给管理员主管
- 在管理后台高亮显示
- 记录SLA违规日志

**SLA时间设置:**
- 紧急: 30分钟首次响应,4小时解决
- 高: 2小时首次响应,24小时解决
- 中: 8小时首次响应,72小时解决
- 低: 24小时首次响应,7天解决

### Q: 内部备注的作用是什么?

A: 内部备注用于管理员之间沟通:

**使用场景:**
- 记录处理思路和进展
- 请求其他管理员协助
- 标记重要信息
- 记录用户背景信息

**特点:**
- 只有管理员可见
- 用户完全看不到
- 不触发用户通知
- 不影响工单状态流转

**最佳实践:**
- 复杂问题记录处理步骤
- 转派时说明处理进展
- 记录特殊用户的注意事项
